# Frontend — Overview

The frontend is a Next.js 16 application using the App Router, hosted on Vercel.

## Route structure

```
app/
├── page.tsx                        # Landing page
├── analysis/
│   └── [id]/
│       ├── page.tsx                # Full report (server component)
│       └── optimize/
│           └── page.tsx            # Budget optimizer (client component)
└── api/
    ├── upload/route.ts             # Receives CSV, fires analysis
    ├── chat/route.ts               # SSE proxy to FastAPI
    ├── optimize/route.ts           # Proxy to /optimize in FastAPI
    └── analysis/[id]/
        ├── status/route.ts         # Analysis status polling
        └── results/route.ts        # Results read (service role)
```

## Main pages

### Landing page (`/`)

- CSV or XLSX upload (XLSX is converted to CSV on the server before upload)
- Preview of detected columns with media channel selection
- Example dataset download

### Report (`/analysis/[id]`)

Server Component — rendered on the server using the service role key.

Sections:
- **Key metrics** — average ROAS, total revenue, number of channels
- **ROAS by channel** — horizontal bars with status (can scale / saturated)
- **Revenue decomposition** — area chart by week showing each channel's contribution
- **Saturation curves** — line per channel showing marginal return
- **AI narrative** — executive report generated by Groq
- **Chat** — Q&A interface about the results

### Optimizer (`/analysis/[id]/optimize`)

Client Component with local state.

- Total budget slider (50% to 200% of current budget)
- 400ms debounce before calling `/api/optimize`
- Comparative bars: current vs. suggested allocation per channel
- Projected revenue uplift

## Upload flow

```
1. User selects file (CSV or XLSX)
2. Frontend detects columns and asks for *_spend channel confirmation
3. POST /api/upload with FormData { file, channels }
   → rate limit by IP (5/hour)
   → size validation (max 10MB)
   → XLSX → CSV conversion if needed
   → upload to Supabase Storage
   → insert into analyses (status: processing)
   → POST /analyze to FastAPI (fire and forget)
4. Redirect to /analysis/[id]
5. Polling at /api/analysis/[id]/status every 2s
6. When status=completed, page loads the data
```

| `FASTAPI_URL` | API routes (server) | Private |

| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Not actively used — may be kept for SDK compatibility | Public |
| `SUPABASE_SERVICE_ROLE_KEY` | API routes (server) | Private |
