# API Reference

The backend is a FastAPI application running on Railway. All endpoints require the `X-Internal-Token` header — requests without it receive `403 Forbidden`. Calls originate exclusively from Next.js API routes.

**Base URL (production):** configured via `FASTAPI_URL` in Vercel.

---

## POST /analyze

Starts an MMM analysis in the background. Returns immediately — processing takes 5–8 min.

### Request body

```json
{
  "analysis_id": "uuid-v4",
  "csv_url": "https://...supabase.co/storage/v1/object/public/csv-uploads/file.csv",
  "channels": ["tv_spend", "search_spend", "social_spend"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `analysis_id` | UUID v4 | ID generated by Supabase on insert |
| `csv_url` | string | Public URL of the file in Supabase Storage |
| `channels` | string[] | Media columns selected by the user |

### Response `200`

```json
{ "analysis_id": "uuid-v4", "status": "processing" }
```

### Validations

- `csv_url` must originate from the project's Supabase instance (validated by prefix)
- `channels` cannot be empty and has a maximum of 20 items
- Each channel must end with `_spend` and be a valid Python identifier

### Background task

FastAPI updates `analyses.step` at each stage:

| Step | Description |
|------|-------------|
| `downloading` | Downloading the CSV from Storage |
| `fitting` | Fitting the MMM model (MCMC) |
| `extracting` | Computing ROAS, contributions, saturation, adstock |
| `optimizing` | Running the budget optimizer (scipy) |
| `narrative` | Generating narrative with Groq |
| `saving` | Persisting results to Supabase |
| `done` | Completed |

At the end, `analyses.status` changes to `completed`. On error, to `failed`.

## POST /optimize

Calculates the optimal budget allocation across channels using `scipy.optimize` (SLSQP).

### Request body

```json
{
  "analysis_id": "uuid-v4",
  "total_budget": 150000.0,
  "current_allocation": {
    "tv_spend": 80000,
    "search_spend": 40000,
    "social_spend": 30000
  },
  "roas": {
    "tv_spend": 1.1,
    "search_spend": 2.66,
    "social_spend": 3.01
  }
}
```

### Response `200`

```json
{
  "analysis_id": "uuid-v4",
  "current_allocation": { "tv_spend": 80000, "search_spend": 40000, "social_spend": 30000 },
  "suggested_allocation": { "tv_spend": 30000, "search_spend": 65000, "social_spend": 55000 },
  "current_revenue_estimate": 338500.0,
  "suggested_revenue_estimate": 401200.0,
  "uplift_percent": 18.5
}
```

## POST /chat

LLM response stream (Server-Sent Events). The model receives the analysis context and the user's message.

### Request body

```json
{
  "analysis_id": "uuid-v4",
  "message": "Why does TV have such a low ROAS?",
  "session_id": "uuid-v4",     // optional — UUID v4; enables per-session rate limiting
  "analysis_context": {
    "roas": { "tv_spend": 1.1, "search_spend": 2.66, "social_spend": 3.01 },
    "contributions": { "tv_spend": 0.18, "search_spend": 0.35, "social_spend": 0.28, "baseline": 0.19 }
  }
}
```

### Response `200` — SSE stream

```
data: Based on the model,

data: the TV channel presents

data: high adstock (slow decay),

data: [DONE]
```

### Rate limits

- `429` if `session_id` has reached 30 messages
- `429` if IP has reached 120 messages/hour
